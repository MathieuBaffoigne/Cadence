# Local API (FastAPI sidecar)

Runs locally alongside the desktop app, backed by SQLite. Source of truth when offline.

## Architecture

Hexagonal (ports & adapters), fully typed:

```
cadence_local_api/
├── domain/
│   ├── models/      # data models (dataclasses) — no framework dependency
│   └── ports/       # ports (ABCs) — no framework dependency
├── application/     # use cases, orchestrate domain ports
├── adapters/        # port implementations (SQLite, ...) — driven/outbound adapters
├── api/
│   ├── routers/     # FastAPI routes, one file per resource — driving/inbound adapter
│   └── dependencies/ # dependency wiring (Depends providers), one file per resource
└── app.py           # composition root (FastAPI app factory)
```

See [ARCHITECTURE.md](ARCHITECTURE.md) for a full walkthrough of each layer with a worked example.

## Tooling

- [uv](https://docs.astral.sh/uv/) for dependencies and running the app
- [ty](https://github.com/astral-sh/ty) for static type checking

## Run

```
uv run alembic upgrade head
uv run uvicorn main:app --port 8756
```

## Schema & migrations

[Alembic](https://alembic.sqlalchemy.org/), migration scripts under `migrations/versions/`. Tables (`clients`, `employees`, `plannings`, `devis`, `factures`) all carry `created_at`/`updated_at` — `updated_at` is what the future cloud sync will diff on.

Migrations are written by hand (`op.create_table(...)`), not autogenerated from ORM models — repositories talk to SQLite directly (see `adapters/`), there is no ORM layer to autogenerate against.

```
uv run alembic revision -m "description"   # new migration
uv run alembic upgrade head                # apply
```

## Test

```
uv run pytest
```

TDD: write the test against `application`/`domain` first, using a fake implementation of the relevant port (see `tests/application/test_health.py`) — no SQLite or FastAPI needed for that layer. Adapters in `adapters/` get their own tests against the real thing (e.g. a temp SQLite file).
